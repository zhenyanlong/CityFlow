# CityFlow：程序化道路规划与交通模拟

> **初稿说明：** 本文件是英文报告的中文对照稿，方便检查内容。提交时应以英文版为主。方括号中的内容必须在提交前补齐。最终版本需要压缩到 LaTeX 模板的 3-4 页内。

## 1. 简短游戏设计文档

### 1.1 游戏概念与核心循环

CityFlow 是一款关于道路规划和交通流动的策略游戏。玩家扮演城市规划者。每局游戏包含三个主要阶段：规划、模拟和结算。

在规划阶段，建筑会被放置在网格上，玩家负责修建主道路网络。每个道路格都会使用有限的共享预算。玩家可以用鼠标左键铺路，用鼠标右键删除道路。道路模型会根据相邻道路自动变化，从而生成直路、弯道、T 字路口和十字路口，不需要玩家手动旋转。

之后，玩家可以启动程序化支路生成器。该系统会尝试把建筑出入口连接到主道路网络上。如果还有足够预算，它也会添加较小的支路。玩家仍然负责关键规划决策，生成器只辅助玩家，而不是代替玩家。

在模拟阶段，道路编辑会被关闭。车辆从建筑生成，使用 A* 寻路，并沿样条线路前往目的地。车辆会在其他车辆后排队，并向繁忙路口申请通行。玩家可以把鼠标放在车辆上，查看它的目的地方向。倒计时结束后，结算界面会显示详细评分报告。

### 1.2 目标、评分与趣味性

CityFlow 不使用简单的胜利或失败界面。游戏目标是创建一个连通而高效的城市。最终得分衡量五个方面：建筑连通性、交通结果、通行效率、预算效率和运行时间。小幅难度倍率用于比较不同规模的地图。

这套评分让玩家同时面对多个目标。连接所有建筑很重要，但仅仅连通还不够。一个路网可能完全连通，却产生很长的路线或严重拥堵。玩家还需要谨慎使用道路预算，并避免让太多车流经过同一个路口。

Random Mode 包含简单、中等和困难三种设置。更高难度会使用更多建筑、更快的车辆生成速度和更长的模拟时间。虽然总预算会略有增加，但平均到每栋建筑的预算会下降，因此压力会逐级提高。

游戏的主要趣味来自控制与意外之间的关系。玩家控制主干道路，而 PCG 系统控制一部分较小的支路。清晰的主干通常会产生整洁结果；较弱的规划可能造成过长支路、预算浪费或建筑断开。这样每座城市都会不同，同时玩家的选择仍然有意义。

### 1.3 叙事与美术风格

游戏采用轻量叙事。玩家是快速发展城市中的规划者。住宅和商业建筑需要接入同一个道路系统。城市由玩家规划的主干道和自动生成的本地道路共同构成。

游戏采用色彩明快的低多边形风格。道路使用简单的灰色形状，建筑使用清晰的体块和容易区分的颜色。车辆较小且颜色鲜艳，便于从俯视镜头观察车流。选择这种风格是因为它在游戏中容易阅读，也符合项目工期。带圆角的地基和倒角人行道让建筑更自然地位于网格上。

游戏还包含数据驱动教程、中英文切换、主音量和音效音量、放置音效、交通警告和分数提示。这些功能可以帮助玩家学习游戏规则。

## 2. 简短技术设计文档

### 2.1 技术概览与架构

CityFlow 使用 Unreal Engine 5.6 开发，目标平台是 Windows。大部分玩法系统使用 C++ 编写，Blueprint 用于配置资源、Widget Blueprint、模型、材质、音频和视觉特效。项目使用了 UMG、Enhanced Input、Niagara、ProceduralMeshComponent、World Subsystem、Data Asset、Sound Class 和 Unreal 原生本地化系统。

项目把主要功能分配给不同管理器：

- `CityFlowGameMode` 控制规划、模拟和结算阶段。
- `GridManager` 保存逻辑网格、占用状态、道路连接和道路预算。
- `LSystemManager` 创建辅助 PCG 道路网络。
- `VehicleManager` 生成车辆、寻找路线并管理交通密度。
- `ScoringManager` 收集结果并生成最终评分明细。
- `CityFlowHUD` 控制主菜单、教程、设置、游戏 HUD、暂停和结算界面。

这些系统通过 Unreal Delegate 处理重要事件。例如，道路变化会更新相邻道路模型和建筑地基；车辆抵达和死亡事件会更新得分与 UI。这样可以减少系统之间的直接引用，使代码更容易测试和修改。

系统架构和游戏循环图源已放在 [diagrams.md](diagrams.md) 中。收到 LaTeX 模板后会导出为矢量图。

### 2.2 网格、建筑和道路

网格使用二维单元数组保存。每个单元记录空地、道路或建筑类型，并保存四方向连接掩码。道路 Actor 读取该掩码，选择正确的模型与旋转。

建筑可以占用多个网格，并以 90 度为单位旋转。出入口保存为本地网格坐标和方向。代码会在建筑旋转后转换这些值，因此道路只能连接到真实出入口。建筑地基通过程序化网格生成；出入口连接道路后，圆角地基形状也会变化。

玩家放置道路时会使用预览 Actor。预览会吸附网格，并根据位置是否合法切换材质。按住鼠标键可以连续铺路，从而更快地完成大型道路规划。

### 2.3 程序化道路生成

项目的原始想法来自关于 PCG 道路生成与 L-system 的 critical review。早期 L-system 原型能够生成有机道路，但不能保证连接所有建筑。即使每个建筑门口附近都有一小段道路，这些道路也可能属于互不连通的不同网络。

最终实现采用混合方法。系统首先寻找主道路连通分量，然后计算未连接出入口的连接路径，并预留这些路径需要的预算。已有道路不产生额外放置成本，因此系统会尽量复用它们。只有剩余预算才可以用于有机支路。候选支路会根据到有效出入口的距离与方向评分，同时保留受控随机性。

这种方法保留了规则生长的视觉特点，同时提供更可靠的玩法结果。每局道路仍然会产生差异，但系统拥有明确目标，并遵守玩家有限的预算。

### 2.4 车辆模拟与反馈

车辆使用 A* 在连通道路格中寻找路线。网格路径会转换为带车道偏移的世界空间样条线。车辆使用前向探测发现其他车辆。T 字和十字路口使用按方向管理的预占机制，避免冲突方向的车辆同时进入繁忙路口。

模拟还包含两种特殊车辆行为。暴走车辆在等待过久后可能忽略路口控制；瞬移车辆在长时间等待后会沿路线向前移动。能力开始时 HUD 会警告玩家。这些事件让严重拥堵更容易被看到，并加强游戏反馈循环。

## 3. Critical Feature 论证

本项目的关键特征是**道路网络的程序化内容生成**，它来自之前对 L-system 的 critical review。[此处加入 critical review 中的 2-3 个引用。] 该功能是 CityFlow 的核心，因为它会影响道路形状、预算使用、车辆路线、拥堵和最终得分。

在图形方面，PCG 使用有限的道路模型生成更多城市布局。自动连接掩码会在每次生成后保持道路视觉连通，因此不需要为每局手工制作独立地图。

在玩法深度方面，玩家需要修建能够引导生成器的主道路。玩家不能只寻找一条最短路线，还要考虑支路空间、建筑入口、共同连通分量和未来交通。连接预算预留也形成了可靠连通与装饰生长之间的取舍。

在玩家参与方面，生成结果提供了有意义的意外。由于候选评分包含受控随机性，相同规划也可能产生小幅差异；同时，连接计划减少了不公平的失败。这种平衡让玩家愿意重复游戏并改进主干布局。

项目也展示了纯 L-system 的限制。局部生成规则擅长产生重复的有机结构，却不会自动理解全局连通性。因此，CityFlow 把局部生长、路径规划和图连通验证结合起来。这是早期原型之后最重要的设计变化。

## 4. 测试与调试总结

项目采用了重复手动游玩、独立游戏和 Editor 构建、运行时日志、作弊命令及可视化调试工具。目前还没有大型正式单元测试套件，这是项目现阶段的限制，需要诚实说明。

主要测试包括：

- 点击与拖拽铺设、删除道路；
- 检查建筑的四种旋转与出入口位置；
- 在不同随机种子和预算下生成道路；
- 确认所有建筑属于同一个道路连通分量；
- 运行简单、中等和困难交通配置；
- 检查路口预占、拥堵、抵达和死亡；
- 测试英文和简体中文 UI；
- 测试有图片和无图片的教程条目；
- 在主要代码变更后编译 Windows 独立游戏目标。

测试过程发现并修复了多个 Bug。最初的 PCG 版本可能在建筑属于不同道路分量时错误报告成功。修复方法是检查真实图连通分量，并预留完整连接路径。旋转后的长方形建筑曾经重复使用旋转尺寸，导致模型与地基使用错误 footprint。修复后，本地模型尺寸与旋转后的逻辑占地被分开处理。随机网格宽高在奇偶变化时还会造成半格偏移；现在网格使用稳定中心锚点，每局开始后也会重绘可视化网格。

另一个问题发生在建筑门口本身是路口时。车辆在路线方向尚未初始化前进入路口碰撞盒，随后被错误标记为已经通过路口，因此会等待一段时间才移动。修复后，车辆直接生成在建筑内部的第一个样条点上，而且只有有效进入路口的车辆才会被标记为已通过。

这些例子展示了迭代式开发过程。可视化调试特别有用，因为许多问题来自逻辑网格和可见世界之间的差异。

## 5. 总结与未来工作

CityFlow 把 PCG 道路研究应用到一款可玩的规划游戏中。它的主要特点是结合玩家主干道路、受 L-system 启发的生长和连通性感知路径规划。项目还把该系统连接到交通模拟、评分、教程、本地化和玩家反馈中。

未来工作应加入针对网格旋转、连通性和道路预算的自动化测试。更多玩家测试数据可以帮助调整难度和交通密度。PCG 系统还可以比较多条有效连接方案，并根据道路形状、通行时间或视觉质量选择结果。
