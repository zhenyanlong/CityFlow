# GDD 文档更新与同步

此 skill 用于在完成一个游戏设计模块后，根据聊天上下文和实际代码/资产内容更新游戏设计文档，并保持中英文版本同步。

## 适用场景

- 用户说 "更新 GDD" 或 "更新游戏设计文档"
- 用户完成了一个游戏设计模块的实现，需要将实际设计情况记录到 GDD
- 游戏设计与 GDD 描述有偏差，需要将文档对齐到实际设计

## 工作流程

### 1. 收集信息

在更新 GDD 之前，需要收集以下信息：

- **聊天上下文**：回顾本次会话中用户讨论的游戏设计点、设计决策、遇到的问题和解决方案
- **代码/资产变更**：使用 `grep` / `SearchCodebase` 等工具查看本次实现涉及的核心文件，了解：
  - 新增/修改了哪些游戏机制、系统、关卡
  - 游戏流程和规则是否有变化
  - 模块之间的交互方式是否有调整
- **资产变更**：查看是否新增或修改了 Blueprint、材质、网格、UI 等资产

### 2. 对比现有 GDD

- 阅读 `GDD.md` 中与本次设计相关的章节
- 找出以下差异：
  - GDD 中描述的设计 vs 实际实现的设计
  - GDD 中缺失的新设计点
  - GDD 中已经过时的描述

### 3. 更新 GDD.md

**必须遵守的原则：**

- **增量修改**：只修改与本次设计相关的章节，不要重写整个文档
- **以实际设计为准**：GDD 描述必须与已实现的设计保持一致
- **保持格式**：沿用 GDD.md 现有的 Markdown 格式规范（标题层级、表格、列表等）
- **记录差异**：如果设计有意偏离了原始计划，在对应描述中简要说明原因（可选）

**常见更新类型：**

| 类型 | 操作 |
|------|------|
| 新增游戏机制/系统 | 在对应章节添加新的子节，描述设计意图和规则 |
| 修改游戏规则 | 更新对应的描述，修改参数和逻辑 |
| 流程变更 | 更新步骤列表，调整顺序或增减步骤 |
| 平衡性/数值调整 | 更新相关章节的具体数值和公式 |

### 4. 同步到 GDD_Chinese.md

更新完 `GDD.md` 后，**必须**同步到 `GDD_Chinese.md`。参考 `gdd-sync` skill 的工作方式，以更新后的 `GDD.md` 为源，逐段翻译变更部分到中文版。

- 如果只有少量章节变更，只翻译变更的章节
- 翻译时保持与 `gdd-sync` skill 相同的规则

### 5. 汇报变更

以简洁的列表形式向用户汇报本次更新了哪些内容：

```
GDD 更新摘要：
- 新增 xxx 游戏机制描述
- 更新 yyy 游戏规则
- 修改 zzz 流程步骤
```
